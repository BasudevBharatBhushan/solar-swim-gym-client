# Changelog v1.1.0 (2026-03-02)

## Authentication & Security Enhancements

### 🛡️ Dual-Layer Token Expiration Handling
Implemented a robust mechanism to handle the 24-hour JWT expiration limit.
- **Layer 1 (Pre-check):** Added manual JWT decoding logic in `apiClient.ts` to verify the `exp` (expiration) claim before sending any request. If expired, the app immediately clears local storage and redirects to the login page.
- **Layer 2 (Post-check):** Added a global interceptor that detects `403 Forbidden` responses with the message `"Invalid or expired token"`. This serves as a server-side fallback for session invalidation.

### 🌐 Tenant Isolation & Location Context
Improved backend data filtering by automatically synchronizing the frontend's active location with the server.
- **Automatic Header Injection:** Both the standard `apiClient` requests and the `upload` method now automatically include the `x-location-id` header sourced from `localStorage`.
- **Seamless Switching:** This ensures that when a staff member or user switches locations in the UI, all subsequent API calls instantly reflect the new tenant context without needing a page refresh.

## Technical Details
- **File Impacted:** `src/services/apiClient.ts`
- **New Helper:** Added `decodeToken(token)` to manually parse JWT payloads without external dependencies (`atob` based).
- **New Helper:** Added `clearAuth()` to centralize the clearing of all authentication-related `localStorage` keys (`token`, `role`, `userParams`, `currentLocationId`, `locations`).
